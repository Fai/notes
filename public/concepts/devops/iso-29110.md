# ISO/IEC 29110

มาตรฐานเกี่ยวกับการจัดการโครงการซอฟต์แวร์ในองค์กรขนาดเล็ก (Software lifecycle management standard for small organizations)

---

## Work Products

### Statement of Work
เอกสารที่ตกลงรายละเอียดงานร่วมกัน

- **Purpose** — วัตถุประสงค์
- **Scope of Work** — ขอบเขตงาน

---

### Project Plan
เอกสารที่ใช้วางแผนและกำหนดกรอบการดำเนินงานอย่างเป็นระบบ

- **Purpose** — วัตถุประสงค์ว่าโครงการทำไปเพื่ออะไร
- **Scope of Work** — ขอบเขตงาน ว่าจะทำอะไร ไม่ทำอะไร
- รายการของงานที่ต้องส่งมอบ
- Project Organization
- **Work Breakdown Structure (WBS)** — งานย่อยแต่ละส่วน
- ระยะเวลาทำงานแต่ละ Task
- **Timeline / Schedule** — เช่น Gantt Chart, Milestone
- **Resource Plan** — Human Resource, Materials, Standards, Tools, Equipment
- **Risk** — ความเสี่ยงที่อาจเกิดขึ้น วิธีจัดการ
- **Version Control Strategy** — การจัดการเวอร์ชั่นของเอกสารและโค้ด
- **Delivery Instructions** — วิธีส่งมอบผลิตภัณฑ์/ระบบ/งาน

---

### Meeting Record
สรุปผลการประชุม เน้นสิ่งที่ได้ตกลงกัน ปัญหาที่หารือ ข้อสรุป และ Action Items

- วัตถุประสงค์
- ประเด็นที่ยังค้างอยู่
- ข้อตกลงร่วมกัน
- การประชุมครั้งถัดไป (วัน เวลา หัวข้อ)

---

### Progress Status Record
บันทึกสถานะความคืบหน้าของโครงการ เปรียบเทียบ Actual vs Planned

---

### Correction Register
ติดตามปัญหาในโครงการที่จำเป็นต้องแก้ไข เช่น Bug, ความผิดพลาดจากกระบวนการ

- ปัญหาเริ่มต้นที่พบ
- แนวทางการแก้ไข
- การดำเนินการแก้ไขที่ได้ทำ
- เจ้าของงานที่รับผิดชอบ

---

### Change Request
บันทึกคำร้องขอการเปลี่ยนแปลงใดๆ ในโครงการ

- วัตถุประสงค์ของการเปลี่ยนแปลง
- สถานะของคำร้องขอ
- ข้อมูลผู้ร้องขอ (ชื่อ เบอร์โทร อีเมล)
- ผู้อนุมัติการเปลี่ยนแปลง
- ระบบที่ได้รับผลกระทบ
- ผลกระทบต่อการใช้งานระบบเดิม
- ผลกระทบต่อเอกสารที่เกี่ยวข้อง (Requirement, Test Case)
- ความสำคัญและวันที่ต้องการใช้งาน

---

### Requirement Specification
บ่งบอกความต้องการของระบบซอฟต์แวร์อย่างชัดเจน ครอบคลุมทั้งด้านฟังก์ชั่น การใช้งาน ความปลอดภัย ประสิทธิภาพ และข้อจำกัด

---

### Test Cases and Test Procedures
คู่มือในการทดสอบ ระบุเงื่อนไข Input/Output/Environment ขั้นตอนและผลลัพธ์ที่คาดหวัง

> คนที่เขียนและรีวิว Test Case ไม่ควรเป็นคนเดียวกัน

- Test ID
- Input / Expected Output
- Environment

---

### Test Report
เอกสารบันทึกผลของการทดสอบ

> ผู้ทดสอบไม่ควรเป็น Developer ที่เขียนโค้ดเอง

- สรุป Defect
- Test Case ที่เกี่ยวข้อง
- ผู้ตรวจพบ
- ความรุนแรง
- ฟังก์ชั่นที่ได้รับผลกระทบ

---

### Traceability Record
เชื่อมโยงความสัมพันธ์ระหว่าง Requirement Specification, Software Design, Software Components และ Test Cases

สถานะของ Applicable Status: `Verified`, `Baselined`, `Updated`

---

### Software User Documentation
เอกสารการใช้งานสำหรับผู้ใช้

---

### Product Operation Guide
เอกสารสำหรับ Admin ผู้ดูแลระบบ

- เกณฑ์การใช้งาน
- วิธีการใช้งาน
- สิ่งแวดล้อม / เครื่องมือประกอบ
- คำเตือนด้านความปลอดภัย
- ขั้นตอนการเตรียมใช้งาน
- คำถามที่พบบ่อย

---

### Acceptance Record
เอกสารยืนยันว่าลูกค้ายอมรับผลส่งมอบของโครงการแล้ว

- รายการที่ส่งมอบ (Deliverables)
- วันที่ลูกค้าได้รับงาน
- Acceptance Criteria
- ปัญหาที่อาจยังค้างอยู่
- ลายเซ็นของลูกค้า

---

### Software Configuration
ชี้จุด Work Products ต่างๆ ที่อยู่ในสถานะพร้อมส่งมอบและได้รับการยอมรับจากลูกค้า

- Requirement Specification
- Software Design
- Traceability Record
- Software Components
- Test Cases and Test Procedures
- Test Report
- Product Operation Guide

---

## Work Process

การทำงานของทั้งสองฝ่ายจะดำเนินไปพร้อมกัน:

- **Project Management** (Project Manager) — ดูแลบริหารโครงการ
- **Software Implementation** (Technical Lead) — พัฒนาซอฟต์แวร์

---

## Appendix — Key Terms

| Term | Definition |
|------|-----------|
| Baseline | เอกสารเวอร์ชั่นล่าสุดที่มีการตรวจสอบแล้ว |
| Change Request | เอกสารที่เสนอการเปลี่ยนแปลง |
| Requirement | ข้อมูลความต้องการของลูกค้า |
| Verify | กระบวนการตรวจสอบความครบถ้วนของงาน |
| Work Product | เอกสารหรือสิ่งที่ได้รับจากการดำเนินกิจกรรม |
