# Work Product
## Statement of Work
เอกสารที่ตกลงรายละเอียดงานร่วมกัน
- Purpose วัตถุประสงค์ 
- Scope of work ขอบเขตงาน
# Project Plan
เอกสารที่ใช้วางแผนและกำหนดกรอบการดำเนินงานอย่างเป็นระบบ โดยครอบคลุมสิ่งที่ต้องทำ ใครทำอะไร กำหนดเวลา และการควบคุมต่างๆ
- Purpose วัตถุประสงค์  ว่าโครงการทำไปเพื่ออะไร มีเป้าหมายอะไร
- Scope of work ขอบเขตงาน ว่าจะทำอะไร ไม่ทำอะไร
- รายการของงานที่ต้องส่งมอบ
- Project Organization
- Work Breakdown Structure (WBS) หรือ Task งานย่อยแต่ละส่วน
- ระยะเวลาทำงานแต่ละ Task
- Timeline / Schedule ในรูปแบบตารางระยะเวลา เช่น Gantt Chart, Milestone
- Resource Plan ที่ใช้เพื่อใช้ประเมินค่าใช้จ่าย ได้แก่ 
	- Human Resource
	- Materials
	- Standards เช่น SOC II
	- Tool เช่น Figma, Miro
	- Equipments ระบุ Hardware ต่างๆที่ใช้ รวมทั้งที่ลูกค้ารับผิดชอบ เช่น Monitor, Server
- Risk ความเสี่ยที่อาจเกิดขึ้น วิธีจัดการ
- Version Control Strategy การจัดการเวอร์ชั่นของเอกสารและโค้ด มีการ backup/recovery ยังไง
- Delivery Instructions จะส่งมอบผลิตภัณฑ์/ระบบ/งานยังไง ต้องมีอะไรบ้าง โค้ด เอกสาร ใครทำอะไรบ้าง

## Meeting Record
สรุปผลการประชุม เน้นสิ่งที่ได้ตกลงกัน ปัญหาที่หารือ ข้อสรุป และ Action Items พร้อมผู้รับผิดชอบและวันที่
- วัตถุประสงค์
- ประเด็นที่ยังค้างอยู่ ระบุเรื่องที่ยังไม่สรุป ต้องตามต่อ
- ข้อตกลงร่วมกัน ระบุ สิ่งที่ทุกฝ่ายเห็นพ้อง
- การประชุมครั้งถัดไป ระบุ วัน เวลา และหัวข้อ​ (ถ้ามี)

## Progress Status Record
บันทึกสถานะความคืบหน้าของโครงการ โดยเปรียบเทียบสิ่งที่เกิดขึ้นจริง (Actual) กับสิ่งที่วางแผนไว้
## Correction Register
ติดตามปัญหาในโครงการที่จำเป็นต้องมีการ ก้ไข เช่น Bug ความผิดพลาดจากกระบวนการ ช่องโหว่จากการตรวจสอบเอกสาร
- ปัญหาเริ่มต้นที่พบ
- แนวทางการแก้ไข
- การดำเนินการแก้ไขที่ได้ทำ
- เจ้าของงานที่รับผิดชอบ
## Change Request
บันทึกคำร้องขอการเปลี่ยนแปลงใดๆในโครงการ ทั้งในด้านฟังก์ชั่น ระบบ กระบวนการ หรือเอกาสร โดยต้องประเมินผลกระทบ และติดตามสถานะของคำร้องขอให้ชัดเจน
- วัตถุประสงค์ของการเปลี่ยนแปลง
- สถานะของคำร้องขอ
- ข้อมูลผู้ร้องขอ ระบุ ชื่อ เบอร์โทร อีเมล์ติดต่อ
- ผู้อนุมัติการเปลี่ยนแปลง
- ระบบที่ได้รับผลกระทบ
- ผลกระทบต่อการใช้งานระบบเดิม
- ผลกระทบต่อเอกสารที่เกี่ยวข้อง ประเมินว่ากระทบกับ Rquirement (Timeline/Cost), Test Case หรือไม่
- ความสำคัญของคำร้องขอ และวันที่ต้องการใช้งาน

##  Traceability Record
ใช้เชื่อมโยงความสัมพันธ์ระหว่าง Requirement Specification, Software Design, Software Components และ Test Cases and Test Procdures เพื่อให้ข้อกำหนดทุกข้อ ได้ถูกออกแบบ พัฒนา ทดสอบแล้วอย่างครบถ้วน โดยสถานะของ Applicable Status ได้แก่ Verified, baselined, updated

## Product Operation Guide
เอกสารให้ข้อมูลจำเป็นในการติดตั้ง และจัดการการใช้งานซอฟต์แวร์อย่างปลอดภัยและถูกต้อง โดยเขียนให้ผู้รับผิดชอบสามารถเข้าใจและปฏิบัติตามได้ สำหรับ Admin ผู้ดูแลระบบ
- เกณฑ์การใช้งาน
- วิธีการใช้งาน

- เครื่องมือประกอบ
- คำเตือนด้านความปลอดภัย
- ขั้นตอนการเตรียมใช้งาน
- คำถามที่พบบ่อย
- แหล่งข้อมูลเพิ่มเติม
- การรับรองด้านความปลอดภัย
- เงื่อนไขการรับประกัน
- ภาษาที่ใช้
## Maintenance Documentation
อธิบายโครงสร้างของซอฟต์แวร์ (Software Configuration) และสิ่งแวดล้อมที่ใช้ในการพัฒนาและปรับปรุงระบบ สำหรับ Devops หรือ Developer
## Verification Results
การตรวจสอบความถูกต้องสำหรับ Quality Assurance ว่ามีการจัดทำหัวข้อใน Work Product และปฏิบัติตาม Process ครบถ้วนตามมาตรฐานหรือไม่
## Validation Results
การตรวจรับจากลูกค้า เช่น เมล์ตอบรับจากลูกค้า การเข้าร่วมประชุมกับลูกค้าและได้รับการตอบรับ การเซ็นรับในเอกสารส่งมอบ
## Acceptance Record
เอกสารยืนยันว่า ลูกค้ายอมรับผลส่งมอบของโครงการแล้ว (เอกสารส่งมอบงาน)
- รายการที่ส่งมอบ (Deliverables)
- วันที่ลูกค้าได้รับงาน
- Acceptance Criteria
- ปัญหาที่อาจยังค้างอยู่ (ถ้ามี)
- ลายเซ็นของลูกค้าเพื่อยืนยันการรับมอบ
